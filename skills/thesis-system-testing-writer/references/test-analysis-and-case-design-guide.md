# 测试分析与用例设计指南

用于提升 6.2 与 6.3 的写作质量。

## 显式需求挖掘

- 建立“来源 -> 需求项 -> 测试点”追踪表。
- 优先覆盖高风险、高频率、关键业务路径对应测试点。
- 区分功能性与非功能性需求（性能、可靠性、可用性、安全性、兼容性）。

## 隐式需求挖掘

- 从领域习惯与用户预期中推断隐藏约束。
- 针对含糊行为向利益相关者提出定向问题。
- 通过探索性测试发现文档未覆盖边界场景。
- 将推断需求转化为可验证测试点，并明确通过/失败判据。

## 多层级需求选择

- 正文不强制详写全部层级，按范围与风险选代表层级展开。
- 明确选择理由（例如：系统测试+回归测试详写，因为发布风险最高）。
- 对省略的同类测试用“同模式”桥接语归纳。

## 用例设计方法选择

关键测试可组合使用：
- 等价类划分
- 边界值分析
- 判定表
- 因果图
- 状态迁移测试
- 场景/用例驱动测试
- 错误推测法

优先级高的测试模块至少采用两种方法，并说明“方法为何匹配该问题”。

## 输出形式建议

- 需求到测试点追踪表
- 带方法标注的代表性测试用例表
- 执行结果汇总表
- 缺陷分布与残留问题汇总
