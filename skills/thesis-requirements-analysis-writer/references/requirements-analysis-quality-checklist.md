# 需求分析质量检查清单

## A. 章节目标是否达成

- [ ] 3.1 明确回答：用户需要什么、为何开发、用户最终得到什么。
- [ ] 3.1 已引出需求分析制品（用例图、活动图、时序图、业务流程图）。
- [ ] 3.2 明确描述了需求导出过程，而非只给结论。
- [ ] 3.2 已说明用户问题、目标、利益相关者与工程化导出方法。
- [ ] 3.2 已给出系统初步说明（业务模型、现状流程、总体要求）。

## B. 功能性需求是否符合经典 SOP

- [ ] 已先给用户需求描述表，字段为 `|需求编号|名称|参与者|说明|`。
- [ ] 已给出系统核心用例图。
- [ ] 已选择核心/突破性用例重点描述，未在正文堆砌全量用例。
- [ ] 重点业务场景补充了动态模型（活动图/顺序图/状态图）。
- [ ] 用例描述字段完整且名称规范，无随意改名。
- [ ] 不是清单式技术报告。

## C. 非功能性需求是否规范

- [ ] 使用软件工程通行分类，不自定义奇异类别。
- [ ] 能定量的指标已定量（如响应时间、故障率、恢复时间）。
- [ ] 每项关键指标给出验证方式或验收口径。
- [ ] 可维护性、可恢复性、可靠性等质量属性有可落地描述。

## D. 可追踪与可读性

- [ ] 功能需求可追溯到利益相关者诉求或访谈证据。
- [ ] 章节逻辑完整：需求导出 -> 功能建模 -> 非功能约束。
- [ ] 图表与正文术语一致（参与者、用例名、流程节点一致）。
- [ ] 图表编号和正文引用一致。
