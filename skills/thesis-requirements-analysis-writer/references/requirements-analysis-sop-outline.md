# 系统需求分析 SOP（第3章）

本 SOP 适用于 3.1-3.4 的规范写作。

## 3.1 系统需求分析概述

本节必须回答三个核心问题：

- 系统开发的用户需要什么？
- 什么样的需求导致了本系统的实施和开发？
- 通过作者开发设计的系统，用户得到什么？

必写内容：

- 业务背景、核心角色、使用场景
- 用户问题与建设动因
- 系统价值与预期收益
- 对需求分析制品的引入说明（用例图、活动图、时序图、业务流程图）

## 3.2 需求导出

本节是后续需求模型的基础，必须说明导出过程。

必写内容：

- 对“用户需要什么”的初步说明
- 利益相关者列表（用户、管理员、运维、管理者、外部系统等）
- 通过访谈/调研导出初步需求的方法与过程
- 用户要解决的问题和要达到的目标
- 引导需求时采用的工程化方法（访谈、观察、研讨、原型反馈、文档分析等）
- 系统初步说明：用叙述/图表概括业务模型、业务流程现状、总体要求

推荐输出：

- 利益相关者表（角色、诉求、关注点）
- 需求导出记录（对象、方法、结论）
- 业务流程现状图或需求导出流程图

## 3.3 功能性需求（分析模型）

目标：在需求导出基础上说明系统的信息域、功能域、行为域。

经典 SOP（严格按顺序）：

1. 先给系统用户需求描述表：

|需求编号|名称|参与者|说明|

2. 给出系统核心用例图（use-case diagram）。
3. 按功能包规模，选择核心用例/突破性业务用例做重点描述，避免在正文罗列全部用例。
4. （可选）对重点业务动态过程，补充 UML 动态模型：顺序图、活动图、状态图。

用例描述要求：

- 用例描述可采用“先给用例图，再给该用例描述表”的方式。
- 全量用例可放附录，不写成清单式技术报告。
- 务必宁缺勿滥，只展开关键用例。

用例描述表：

|用例名称|| |用例描述|| |主执行者|| |优先级|| |触发条件|| |后置条件||
|基本事件流|| |异常事件流||

## 3.4 非功能性需求

本节描述“满意系统”的质量特征与约束条件。

必写要求：

- 采用软件工程规范化分类，不能随意自定义名称。
- 能量化的指标必须量化，建议以图表呈现。
- 指标需可验证，给出验收口径。

建议覆盖维度：

- 可靠性（故障率、可恢复性、可维护性）
- 性能（响应时间、吞吐、并发）
- 安全性（权限控制、审计、数据保护）
- 可用性/易用性
- 环境与部署约束（开发环境、运行环境）
