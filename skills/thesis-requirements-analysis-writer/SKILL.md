# 论文需求分析写作器

用于完整撰写或重写第3章“系统需求分析”。

## 工作流

1. 明确系统边界与章节目标：回答“用户要什么、为何开发、上线后得到什么”。
2. 收集需求导出材料：用户问题、目标、利益相关者列表、访谈记录、业务现状流程。
3. 按 `references/requirements-analysis-sop-outline.md` 写作 3.1-3.4。
4. 在功能性需求中执行经典
   SOP：先给需求描述表，再给核心用例图，再选择核心/突破性用例做描述。
5. 用 `references/requirements-analysis-quality-checklist.md` 逐项自检并修订。

## 输入收集（最低要求）

- 系统名称、业务场景、目标用户
- 用户当前问题、建设目标、预期收益
- 利益相关者及访谈/调研证据
- 业务流程现状与约束条件
- 候选功能需求及优先级
- 非功能性指标（可靠性、性能、安全、可恢复性、可维护性、环境约束）

输入不足时：先写明假设，再产出初稿，并单列“待补材料”。

## 写作硬规则

- 必须体现需求分析文档：静态用例图；动态活动图、时序图、业务流程图（按场景选用）。
- 3.2 需求导出必须说明“如何导出”，不是只给结果。
- 3.3 功能性需求必须先给需求描述表：`|需求编号|名称|参与者|说明|`。
- 3.3 必须出现系统核心用例图，并只展开核心/突破性业务用例（宁缺勿滥）。
- 用例描述使用标准字段，不得随意改名。
- 3.4 非功能性需求必须使用软件工程通行分类，能定量就定量，并给验证口径。
- 避免把章节写成全量清单式技术报告。

用例描述表模板见 `references/use-case-description-table-template.md`。

## 重写模式

当用户给出原文时：

1. 诊断缺口：需求导出是否缺失、功能建模是否清单化、非功能是否不可验证。
2. 保留有效事实，重写空泛段落。
3. 输出“重写后正文 + 待补材料清单（访谈证据/图表/量化指标）”。
